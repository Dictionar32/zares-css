## Part 1: ARIA vs Variants Clarification

### User Question
**"ARIA buat apa? Kan ada variant untuk handle visual states?"**

### Answer Provided
ARIA dan Variants adalah **complementary**, bukan competing:

| Aspek | Variants | ARIA |
|-------|----------|------|
| Purpose | Visual/styling states | Semantic/accessibility info |
| User | Sighted users | Screen reader users |
| Example | `size: { sm, md, lg }` | `role: 'button'`, `aria-pressed` |
| Rendered | CSS classes | HTML attributes |
| Layer | Visual appearance | Accessibility tree |

### Key Insight

```typescript
// ❌ Variants only (visual but not accessible)
<Button primary>Dark Mode</Button>
// Screen reader: "Dark Mode button" ← doesn't know it's a toggle

// ✅ Variants + ARIA (visual + accessible)
<Button 
  intent="primary"        // Visual via variants
  aria-pressed={isActive} // Semantic via ARIA
>
  Dark Mode
</Button>
// Screen reader: "Dark Mode toggle button, pressed"
```

## Part 2: Dev Server Warnings Resolution

### Problem Statement

When running `npm run dev` in next-js-app, 3 warnings appeared:

```
⚠ Turbopack root inference warning
ℹ Native binding unavailable (normal in dev)
⚠ Hydration mismatch warning
```

### Analysis & Root Causes

#### 1. Turbopack Root Warning
- **Cause**: Multiple `package-lock.json` files in monorepo
- **Impact**: Configuration warning only
- **Severity**: Low

#### 2. Native Binding Message
- **Cause**: Native module (.node binary) not available in JavaScript environment
- **Impact**: Fallback to TypeScript implementation
- **Severity**: Informational (NOT an error)

#### 3. Hydration Mismatch Warning
- **Cause**: Theme initialization script modifies `<html>` style attribute
- **Impact**: Server-rendered state might differ from client state
- **Severity**: Medium (needs suppression flag)

---

## Part 3: Solutions Implemented

### Fix #1: Turbopack Root Configuration

**File**: `examples/next-js-app/next.config.ts`

```typescript
// Added:
const nextConfig: NextConfig = {
  reactCompiler: true,
  experimental: { /* ... */ },
  turbopack: {
    root: __dirname,  // ← Explicitly point to app directory
  },
};
```

**Result**: ✅ Turbopack warning eliminated

---

### Fix #2: Hydration Mismatch Suppression

**File**: `examples/next-js-app/src/app/layout.tsx`

```typescript
// BEFORE
<html lang="en">
  <body className="...">

// AFTER
<html lang="en" suppressHydrationWarning>
  <body className="..." suppressHydrationWarning>
```

**Rationale**:
- Theme script intentionally modifies `<html>` style attribute
- Server state might differ from client (localStorage dependent)
- Suppression flag tells React: "This is expected, don't warn"
- Safe because actual content is identical (only styling differs)

**Result**: ✅ Hydration mismatch warning eliminated

## Key Takeaways

### 1. ARIA and Variants are Complementary
- Variants = Visual (CSS)
- ARIA = Semantic (Accessibility)
- Both needed for true accessibility

### 2. Hydration Mismatch is Expected in Some Cases
- Theme initialization intentionally modifies DOM
- Safe to suppress when content is identical
- Only styling (CSS variables) differ

### 3. Monorepo Configuration Matters
- Multiple lockfiles need explicit configuration
- Turbopack can't auto-detect in complex setups
- Explicit `root: __dirname` solves it

### 4. Dev Server Messages vs Errors
- Not all console messages are problems
- Native binding message is informational
- Distinguish between warnings, errors, and info
