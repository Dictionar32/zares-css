## PHASE 1: Initial Scanning (Rust Engine)

### Apa yang Terjadi

```javascript
// withTailwindStyled.ts line ~355-400
const result = scanWorkspace(srcDir)
// Rust engine scans 81 files in src/
// Extracts ALL Tailwind class strings: "px-4", "bg-blue-600", etc.
```

**Input**: semua `.ts`, `.tsx`, `.js`, `.jsx` files di `src/`

**Process** (Rust engine):
1. Parse setiap file menggunakan Rust AST parser (oxc)
2. Cari string literals yang terlihat seperti Tailwind classes
3. Extract class names dari template literals:
   ```tsx
   tw.button`px-4 py-2 bg-blue-600`  // → ["px-4", "py-2", "bg-blue-600"]
   ```
4. Extract dari object configs:
   ```tsx
   const Button = tw.button({
     base: "inline-flex px-4",     // → ["inline-flex", "px-4"]
     variants: { ... }
   })
   ```
5. Filter false positives:
   - Computed values: `top-[205.64px]` (float precision, generated)
   - Subcomponent keys: `"header:topBar"` (not a Tailwind class)
   - Very large values: `top-[99999px]` (obviously computed)

**Output**: `result.uniqueClasses` = deduplicated array of 1000+ class names

### Files Generated

**`_initial-scan.css`** (3500+ lines)
```css
/* tw-classes: initial scan — auto-generated by withTailwindStyled */
@layer utilities {
  .pointer-events-auto { pointer-events: auto; }
  .pointer-events-none { pointer-events: none; }
  .px-4 { padding-left: calc(var(--spacing) * 4); padding-right: calc(var(--spacing) * 4); }
  .py-2 { padding-top: calc(var(--spacing) * 2); padding-bottom: calc(var(--spacing) * 2); }
  .bg-blue-600 { background-color: var(--color-blue-600); }
  /* ... 3000+ more ... */
}
```

**Kenapa begini?**
- Tailwind v4 membutuhkan `@source` directives atau CSS class list untuk tahu class apa yang di-gunakan
- Scanner extracts semua classes yang MUNGKIN di-gunakan
- Tailwind PostCSS plugin melihat CSS ini sebagai "I want these classes"
- Tailwind generates seluruh CSS untuk class-class tersebut
- Static analysis → deterministic → zero guessing di runtime

### `_start.txt`

```
1783014112750
```

Timestamp kapan dev server di-start. Digunakan oleh Turbopack loader untuk detect:
- "Apakah ini dev server yang sama atau restart baru?"
- Jika restart, clear cache dari session lama

## PHASE 2: Route Attribution & CSS Splitting

### Konsep

Next.js App Router menggunakan file structure untuk routing:
```
app/
  layout.tsx           → shared layout (all routes)
  page.tsx             → route /
  learn/
    layout.tsx         → shared layout for /learn/*
    page.tsx           → route /learn
    mentor/
      page.tsx         → route /learn/mentor
```

Engine perlu tahu: **which component is used by which route**

```
/ route uses: Button, Card, Header, Footer
/learn route uses: Button, Card, Header, SideNav, CodeBlock
/learn/mentor route uses: Button, MentorCard, Timeline
```

Mengapa? **Route-specific CSS optimization**:
- `/` tidak perlu `MentorCard` styles → skip dari bundle
- `/learn/mentor` tidak perlu `CodeBlock` styles → skip dari bundle
- Setiap route hanya load CSS yang benar-benar di-gunakan

### Implementation

```typescript
// withTailwindStyled.ts: route graph building
const routeClassBuckets = buildRouteClassBuckets(srcDir, files)
// Output example:
{
  "/": ["Button", "Card", "Header"],
  "/learn": ["Button", "Card", "Header", "SideNav"],
  "/learn/mentor": ["Button", "MentorCard", "Timeline"],
  "__global": ["Layout", "Nav"]  // shared by all routes
}
```

**Algorithm** (`routeGraph.ts`):
1. Scan `app/**/*.tsx` untuk find route entry points (`page.tsx`, `layout.tsx`)
2. Build import graph menggunakan regex static analysis:
   ```
   import Button from "@/components/Button"
   import { Card } from "@/components"
   require("./helper")
   import("./dynamic")  // literal string only
   ```
3. BFS dari setiap route entry
4. Collect semua files reachable dari route itu
5. Map files → classes (dari phase 1)
6. Attribute classes → route

**Fallback**: file yang unreachable atau shared → `__global` bucket

### Files Generated

**`css-manifest.json`** (dalam memory atau `.next/static/`)
```json
{
  "/": "tw-route-abc123.css",
  "/learn": "tw-route-def456.css",
  "/learn/mentor": "tw-route-ghi789.css",
  "__global": "tw-global.css"
}
```

**`tw-route-abc123.css`**
```css
/* Only classes used by / route */
.inline-flex { display: inline-flex; }
.items-center { align-items: center; }
.px-4 { padding: ... }
/* ... etc ... */
```

## PHASE 3: State Rules Pre-Generation

### Konsep

Component dapat punya `states`:
```tsx
const Button = tw.button({
  base: "px-4 py-2 bg-blue-600",
  states: {
    loading: "opacity-60 cursor-wait",
    disabled: "opacity-50 cursor-not-allowed",
    fullWidth: "w-full"
  }
})

// Runtime usage:
<Button loading>Processing...</Button>
// Renders: <button class="..." data-loading="true">
// CSS selector matches: .tw-s-b35937[data-loading="true"] { opacity: 60%; cursor: wait; }
```

### Problem That Needed Solving

**Old approach (runtime injection)**:
```typescript
// Runtime: when Button mounts with loading=true
const styles = `
  .tw-s-b35937[data-loading="true"] {
    opacity: 60%;
    cursor: wait;
  }
`
const style = document.createElement("style")
style.textContent = styles
document.head.appendChild(style)  // ⚠️ Runtime overhead + potential flicker!
```

**New approach (pre-generated)**:
```css
/* _tw-state-static.css (pre-generated at build time) */
.tw-s-b35937[data-loading="true"] {
  opacity: 60%;
  cursor: wait;
}
```

Runtime hanya set `data-loading="true"` attribute → CSS instantly applies ✨

### Implementation

```typescript
// staticStateExtractor.ts
const stateRules = extractComponentStates(srcDir)
// Scan 81 files → find 182 components
// Extract `states` config dari setiap component
// Pre-generate CSS untuk setiap state

// Generate hash untuk selector determinism
const hash = componentIdHash(componentName, stateName)
// Output:
{
  selector: ".tw-s-b35937",
  stateName: "loading",
  declarations: "opacity: 60%; cursor: wait;",
  rule: ".tw-s-b35937[data-loading=\"true\"] { opacity: 60%; cursor: wait; }"
}
```

## PHASE 4: Loader Integration

### Webpack Loader (webpackLoader.ts)

Used when: `next build --webpack` flag (or Next.js configured with Webpack)

```typescript
export default function webpackLoader(this: LoaderContext, sourceCode: string) {
  // Per-file: extract Tailwind classes
  const ast = parseTypeScript(sourceCode)
  const classes = extractTwClasses(ast)
  // Hash untuk deterministic output
  const componentHash = hashComponent(filePath, classes)
  // Register per file
  registerFileClasses(filePath, classes, componentHash)
  // Return unchanged source — loaders hanya extract, tidak modify
  return sourceCode
}
```

### Turbopack Loader (turbopackLoader.ts)

Used when: `next dev` or `next build` (Turbopack is Next.js 16+ default)

```typescript
// Per file — extract Tailwind classes
const classes = extractClasses(resourcePath, sourceCode)
// Deterministic hash untuk cache-busting
const hash = hashFileClasses(resourcePath, classes)
// Register dengan route attribution
registerFileClasses(resourcePath, classes, hash)
// Return unchanged source
return sourceCode
```

**Why both loaders?**
- Webpack: explicit flag, for advanced users
- Turbopack: default, faster bundling, custom loader API
- Both extract same data, feed into route attribution

### Cycle Detection

```
_start.txt = 1783014112750      (written by withTailwindStyled at config load)
_cycle.txt = (per-loader write, per Turbopack compile cycle)

turbopackLoader.ts:
  • Baca _start.txt
  • Compare dengan _cycle.txt
  • If same: reuse cache
  • If different: this is new dev server, clear cache
```

## How CSS Gets to Browser

### Step 1: globals.css

```css
@import "tailwindcss";
@import "./.next/tw-classes/_tw-state-static.css";
/* or auto-injected by withTailwindStyled */

:root { --background: #f5f7fb; }
[data-theme="dark"] { --background: #070b16; }

@theme inline {
  --color-background: var(--background);
}
```

### Step 2: Tailwind CSS v4 Processing

Tailwind PostCSS plugin sees:
- `@import "tailwindcss"` → generate all Tailwind utilities
- Classes in `_initial-scan.css` → filter to only what's used
- Custom `@theme inline` → bridge CSS variables to design system

Output: full `dist/index.css` (oder route-specific CSS if splitting enabled)

### Step 3: TwCssInjector (Optional, for Route Optimization)

```tsx
// layout.tsx
<head>
  <TwCssInjector />
</head>
```

**Server component** that:
- Reads `css-manifest.json`
- Looks up current route
- Injects `<style>` tag with route-specific CSS inline
- Benefit: no extra HTTP request, no FOUC

Falls back gracefully if manifest missing (legacy builds)

### Step 4: Runtime

When component renders:
```tsx
<Button loading>Process</Button>
// Browser sees: <button class="..." data-loading="true">
// CSS selector matches: .tw-s-b35937[data-loading="true"]
// Styles applied instantly ✨
```

## Technical Details

### Component Hash Generation

```rust
// native/src/napi_bridge_types.rs
fn compute_component_id(name: &str, state_name: Option<&str>) -> u32 {
  let mut hasher = /* FxHasher or similar */
  hasher.write(name.as_bytes())
  if let Some(state) = state_name {
    hasher.write(b":")
    hasher.write(state.as_bytes())
  }
  hasher.finish() as u32 & 0xFFFFFF  // 24-bit hash
}

// Example output:
// Button + disabled → .tw-s-b35937
// H2 + copied → .tw-s-93c530
// StateButton + loading → .tw-s-54d869
```

Deterministic: same input = same hash, every time ✨

### Class Name Validation

```typescript
// withTailwindStyled.ts: filter false positives
const isValidTwClass = (cls: string): boolean => {
  // Skip variant prefixes that don't exist in Tailwind
  if (cls.includes(":")) {
    const prefix = cls.split(":")[0]
    if (!VALID_VARIANT_PREFIXES.has(prefix ?? "")) return false
  }

  // Skip computed values (float precision)
  if (/\[[\d]+\.[\d]{2,}(?:px|rem|em|vh|vw|%)\]/.test(cls)) return false

  // Skip values that are impossibly large (>9999px)
  if (/\[[\d]{5,}(?:px|rem|em)?\]/.test(cls)) return false

  return true
}
```

Catches ~95% of false positives, keeps _initial-scan.css reasonable size
